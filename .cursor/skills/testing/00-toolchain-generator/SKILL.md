# Toolchain Generator

---

## When to Trigger

- User says: "generate test toolchain", "initialize testing", "set up tests"
- A new project is created and Spec/PRD is confirmed
- The project's tech stack changes significantly (e.g., migrated from AppKit to SwiftUI)
- A testing methodology Skill references `.cursor/skills/testing/toolchain/*-[platform].md` but the file does not exist

---

## Step-by-Step Execution

### Step 1 — Detect Tech Stack

Read the project's Spec, PRD, or codebase to identify:

```
Platform:        [macOS / iOS / Web / Chrome Extension / ...]
Language:        [Swift / TypeScript / JavaScript / ...]
UI Framework:    [AppKit / SwiftUI / UIKit / React / Vue / ...]
Build System:    [Xcode / Webpack / Vite / ...]
Package Manager: [SPM / CocoaPods / npm / pnpm / ...]
Backend/API:     [Supabase / Firebase / REST / GraphQL / none / ...]
State Manager:   [Zustand / Redux / @Observable / ...]
Key Dependencies: [StoreKit / ScreenCaptureKit / Radix UI / ...]
```

### Step 2 — Confirm with User

Present the detected stack and ask for confirmation:

```
🔍 Detected tech stack:
   Platform:     macOS
   Language:     Swift
   UI Framework: AppKit (pure code layout, no SwiftUI)
   Build System: Xcode (.xcodeproj)
   ...

Is this correct? Should I generate testing toolchain configs based on this?
```

Do NOT proceed without user confirmation.

### Step 3 — Generate 6 Toolchain Files

Generate the following files in `.cursor/skills/testing/toolchain/`:

```
unit-[platform].md
ui-[platform].md
snapshot-[platform].md
e2e-[platform].md
performance-[platform].md
accessibility-[platform].md
```

Where `[platform]` is a short identifier: `swift`, `react-ts`, `vue-ts`, etc.

### Step 4 — Report

```
🛠️ Toolchain Generation Complete
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Platform: [platform]
Files generated in .cursor/skills/testing/toolchain/:
  ✅ unit-[platform].md
  ✅ ui-[platform].md
  ✅ snapshot-[platform].md
  ✅ e2e-[platform].md
  ✅ performance-[platform].md
  ✅ accessibility-[platform].md

These files are now linked to your testing methodology Skills.
```

## Platform Reference Guide

Use the following as reference when generating toolchain files. Adapt based on the actual detected tech stack.

### Swift (macOS / iOS)

| Test Type | Framework | Key Tool | Notes |
|-----------|-----------|----------|-------|
| Unit | XCTest | `@testable import`, `XCTestCase` | Protocol-based DI for mocking singleton Services |
| UI | XCUITest | `XCUIApplication`, `XCUIElement` | Launch arguments for test configuration |
| Snapshot | XCTest + swift-snapshot-testing | `assertSnapshot(of:as:)` | Record mode for baseline, pixel tolerance for CI |
| E2E | XCUITest | Full `XCUIApplication` flow | Separate test scheme, longer timeout |
| Performance | XCTest | `measure {}`, `XCTMetric` | Set baselines, max deviation 10% |
| Accessibility | XCTest + Accessibility Inspector | `XCUIElement.isAccessibilityElement` | VoiceOver labels, keyboard navigation |

### TypeScript + React (Chrome Extension / Web)

| Test Type | Framework | Key Tool | Notes |
|-----------|-----------|----------|-------|
| Unit | Vitest | `vi.mock`, `vi.fn` | Mock chrome.* APIs, Supabase client, Zustand stores |
| UI | Vitest + @testing-library/react | `render`, `screen`, `userEvent` | Test behavior not implementation; query by role/label |
| Snapshot | Vitest | `toMatchSnapshot()`, `toMatchInlineSnapshot()` | Commit snapshots to git, review diffs carefully |
| E2E | Playwright | `chromium.launch`, extension loading | Load unpacked extension, test full user flows |
| Performance | Lighthouse CI + web-vitals | `getCLS`, `getFID`, `getLCP` | Budget thresholds for Core Web Vitals |
| Accessibility | Vitest + axe-core / @axe-core/react | `axe.run()`, WCAG 2.1 AA | Auto-audit all rendered components |

### Other Platforms (Vue, Angular, iOS UIKit, Flutter, etc.)

If the detected platform is not listed above:
1. Research the best-practice testing tools for that platform
2. Follow the same template structure
3. Ask the user to confirm the chosen tools before generating

## Multi-Platform Projects

If a project contains multiple platforms (e.g., shared backend + web frontend + mobile):

1. Generate SEPARATE toolchain files per platform
2. Use clear platform identifiers: `swift`, `react-ts`, `vue-ts`, `flutter`, etc.
3. Report all generated files grouped by platform

```
🛠️ Toolchain Generation Complete (Multi-Platform)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Platform: swift (macOS app)
  ✅ unit-swift.md
  ✅ ui-swift.md
  ...

Platform: react-ts (Chrome extension)
  ✅ unit-react-ts.md
  ✅ ui-react-ts.md
  ...
```

## Regeneration Rules

Regenerate toolchain files when:
- Major framework migration (e.g., AppKit → SwiftUI)
- Build system change (e.g., Webpack → Vite)
- New critical dependency added (e.g., adding Supabase to a project that had no backend)
- User explicitly requests: "regenerate test toolchain"

When regenerating:
1. Back up existing toolchain files (rename with `.bak` suffix)
2. Generate new files
3. Report what changed
